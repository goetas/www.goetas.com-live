Why JavaScript Developers Keep Avoiding Dependency Injection (And Why That’s a Problem)sss

Most articles highlight how Dependency Injection simplifies testing and improves modularity and maintainability. In this post I will show how DI accelerates development and boosts developer experience

Dependency Injection: Why it matters not only for testing
2025-06-01

In my previous post, I discussed Dependency Injection (DI) and how it improves overall developer experience (DX).

In this post I will focus more on JavaScript developers and some of the most common excuses to avoid it.

Dependency injection isn’t a new concept. It’s been around for decades in other ecosystems — Java, .NET, PHP and others. It’s not magic, it is a way to organize code so that dependencies are provided from the outside instead of hardcoded inside.

Let’s start from the start

JavaScript started as a frontend language, in the browser, in that world, the code needs to be transferred over the network. In that context, there are (were?) valid reasons to avoid DI frameworks:

  • Bundle size matters, a centralized DI container can bloat your code as it might need to include all dependencies (and the workarounds to avoid that are often complex)
  • Bundle chunking / tree shaking makes it harder to split dependencies
  • Javascript minification can remove DI metadata (such as variable names), breaking reflection-based DI frameworks
  • Javascript decorators are not yet mature, making it hard to use them for DI (when compared to languages like Java or PHP where annotations are widely used)

These constraints led Javascript developers to a mindset of keeping things simple as much as possible.

But then Node.js came along

Suddenly, JavaScript could be run server-side, and it pawed the way for services, APIs, background jobs.

On the server, the constraints changed, is no network to worry about. By consequence:

  • No need to worry about bundle size
  • Minification is not a needed
  • Disk space and RAM are cheap

All the valid reasons to avoid DI when running your code in the browser disappeared in the server context.

Unfortunately, the mindset did not adapt, "We like to keep things simple" was still the mantra.

The "Simple" Codebase Myth

But what ends up happening in these “simple” codebases? For a small library or a one-endpoint service, sure, it might work. But what happens to a codebase that grows to 10k lines, 50k lines, or more?

Any large Javascript project is made of tens (hundreds)? files that all import the same services manually, wiring dependencies between functions/classes in the most creative way. Patching things globally because there’s no clean way to swap dependencies during testing. Suddenly, 30% of your code is not business logic — it’s wiring. And when something needs to change (like adding a dependency to an existing function/class), you're hunting down every place it was directly used. How is that simple? On-boarding new developers? Good luck.

JavaScript with and without DI

JavaScript is a language that allows multiple paradigms, and it can be used in a functional, object-oriented, or procedural style. When using and object-oriented style, JavaScript has a classes and decorators, which can be used to implement Dependency Injection in a very similar way to what shown in my previous post.

Let's consider a simple example of a service that reads a configuration file and returns a server URL.

Without DI

A plain Typescript implementation without DI might look like this:

// config-service.ts (a service that reads a configuration file and returns a server URL)
import fs from 'fs'

export function getServerURL(fileName: string): string {
  const json = JSON.parse(fs.readFileSync(fileName, 'utf-8'));
  return json.url;
}

// main.ts (the main entry point of the application)
import { getServerURL } from './config-service';
const serverURL = getServerURL('config.json');

Seems simple, right? But:

  1. How to test getServerURL without actually having to create a file to read by it?
  2. What if we need to the currently logged-in user in the getServerURL function? (to provide a different URL based on the user?)

- getServerURL might be used in dozens of places, and we would need to change all of them to pass the user information. - or we would have to find a way to extract the user information from some global state, which is not a good practice.

None of the two problems have a good solution in the above code. The problem n.2 is the most common scenario in any evolving company.

With DI

Many developers prefer to use JavaScript in a functional style, a better version of the above cod look like this:

// config-service.ts (a service that reads a configuration file and returns a server URL)
import fs from 'fs'

type Reader = (fileName: string) => string;
type ServerUrlProvider = (fileName: string) => string;

export const fileReader: Reader = (fileName: string): string {
  return  fs.readFileSync(fileName);
}

export const getServerURL: ServerUrlProvider = (fileReader: Reader, fileName: string): string {
    const json = JSON.parse(fileReader(fileName));
    return json.url;
}

// main.ts (the main entry point of the application)
import { getServerURL } from './config-service';
const serverURL = getServerURL(fileReader, 'config.json');

This version is a bit more flexible, as it allows to pass a custom fileReader function, improving testability.

However,the problem n.2 remains. We need to hunt down all the places where getServerURL was used and change them to pass them additional dependencies when needed.

We could make it better:

// file-reader.ts (a simple file reader implementation)
import fs from 'fs'

type Reader = (fileName: string) => string;
type ServerUrlProvider = (fileName: string) => string;

export const fileReader: Reader = (fileName: string): string {
  return fs.readFileSync(fileName);
}
// server-url.ts (a service that reads a configuration file and returns a server URL)
export function createGetServerUrl(fileReader: Reader): ServerUrlProvider {
  return function (fileName: string): string {
    const json = JSON.parse(fileReader(fileName));
    return json.url;
  };
}

// di.ts (Dependency Injection Container definition)
import { createGetServerUrl } from './config-service';
import { fileReader } from './file-reader';

export const getCreateGetServerUrl = () :ServerUrlProvider => createGetServerUrl(fileReader);


// main.ts (the main entry point of the application)
import { getCreateGetServerUrl } from './di'

export const getServerURL = getCreateGetServerUrl();

const serverURL = getServerURL('config.json');

This looks much better. We have decoupled the dependencies from the function itself. The function creation is now separated from its invocation.

  1. Testing getServerURL is straightforward. You can call createGetServerUrl with a mock implementation of Reader that returns a predefined JSON string.
  2. Adding additional dependencies, such as the currently logged-in user, is now straightforward. The only place that needs to be updated is the service creation di.ts.

You might ask why do we getCreateGetServerUrl instead of exporting createGetServerUrl directly. By doing that we delay the dependency resolution until the function is actually used in the application since some dependencies might not be ready yet.

// file-reader.ts (a simple file reader implementation)
import fs from 'fs'

type Reader = (fileName: string) => string;

@injectable()
export const fileReader: Reader = (fileName: string): string {
  return fs.readFileSync(fileName);
}
@injectable()
export function createGetServerUrl(@inject() fileReader: Reader): ServerUrlProvider {
  return function (fileName: string): string {
    const json = JSON.parse(fileReader(fileName));
    return json.url;
  };
}

// di.ts (Dependency Injection Container definition)

export const getServerURL = DI.get('createGetServerUrl');


// main.ts (the main entry point of the application)
import { getServerURL } from './di';
const getServerURL = DI.get('ServerUrlProvider');
const serverURL = getServerURL('config.json');

Seems simple, right? But:

  1. How to test getServerURL without actually having to create a file to read by it?
  2. What if we need to the currently logged-in user in the getServerURL function? (to provide a different URL based on the user?)
    • getServerURL might be used in dozens of places, and we would need to change all of them to pass the user information.
    • or we would have to find a way to extract the user information from some global state, which is not a good practice.

None of the two problems have a good solution in the above code. The problem n.2 is the most common scenario in any evolving company.

With DI

When using a DI framework as InversifyJS, we might do something like this:

// reader.ts (define our Reader interface)
export interface Reader {
  readFile(fileName: string): string;
}

// config-service.ts (and example service that uses the Reader interface to read a configuration file)
import Reader from './reader'
import { inject, injectable } from 'inversify';

@injectable()
export class ConfigService {
    constructor(@inject('Reader') private fileReader: Reader) {
    }

    getServerURL(fileName: string): string {
        const json = JSON.parse(fileReader.readFile(this.fileName));
        return json.url;
    }
}

You can see that the ConfigService class depends on a Reader interface, but is totally unaware of the actual implementation of that interface.

// file-system-reader.ts 
import fs from 'fs'
import { injectable } from 'inversify';

@injectable()
export class FileSystemReader implements Reader {
  readFile(fileName: string): string {
    return fs.readFileSync(fileName, 'utf-8')
  }
}

This is just one possible implementation of the Reader interface, you can have multiple implementations, such as reading from a database or an API or being just a mock file system.

// di.ts (Dependency Injection Container definition)
import { Container, inject, injectable } from 'inversify';
import {ConfigService} from './config-service'
import {Reader} from './reader'
import {FileSystemReader} from './file-system-reader'

const container: Container = new Container();

container.bind<Reader>('Reader').to(FileSystemReader);
container.bind<ConfigService>('ConfigService').to(ConfigService);

export default container;

This part is where the Dependencies are defined. It is important to notice that at this point we are not worrying about resolving the dependencies, we are just defining them. Order does not matter either.

This part is fairly "mechanical", in frameworks like SpringBoot (Java) or Symfony (PHP), this file is not needed as it is done automatically. In JavaScript, needs to be done manually, hopefully, some frameworks will provide this functionality in the future. NestJS or Angular are already heading in this direction.

// main.ts (the main entry point of the application)
import container from './di'

const configService: ConfigService = container.get('ConfigService');

const serverURL = configService.getServerURL('config.json');

And finally we call the getServerURL to get the server URL.

This version might look a bit verbose, but it it shines when complexity increases.

  1. Testing the ConfigService is now straightforward. You can create a mock Reader implementation that returns a predefined JSON string
  2. If you need the currently logged-in user you can simply inject a CurrentUserProvider service. You do not need to worry about finding all the places where ConfigService was used, nor you need to worry about the dependencies needed for the CurrentUserProvider service.
// config-service.ts 

@injectable()
export class ConfigService {
    constructor(
        @inject('Reader') private fileReader: Reader,
        @inject('CurrentUserProvider') private currentUserProvider: CurrentUserProvider,
    ) {
    }

    // ...
}

If the CurrentUserProvider was already registered in the DI container by some of our colleagues, the work is done. No need to worry about finding all the places where ConfigService was used.

Of course, if we did not have already CurrentUserProvider we would need to implement and register it, but that would have been done anyway, even if we were not using DI.

JavaScript Is Special, but not that much

The idea that JavaScript is some “special case” where dependency injection doesn’t apply is flawed. Every language thinks that, until it doesn’t.

Go developers used to hate DI. Then Uber built Dig. Rust has shaku. Python has Dependency Injector. Even dynamic languages are using it now—because it helps.

JavaScript developers moved to the backend but brought along their frontend baggage. The “we don’t need that” attitude worked when your code ran in the browser. It doesn’t scale when you’re building services, APIs, and large systems.

You don’t have to go full Spring or Angular. But pretending DI is “too complex” while manually wiring everything yourself is just denial.

It’s time to grow up the stack. Use the right patterns for the job. And stop pretending dependency injection is some enterprise-only monster.

It’s just architecture. And it makes your life easier.

Conclusion

Conclusion (2.0)

This post is certainly a bit of a rant (and provocative). The JavaScript community it is the biggest community in the world, and at this point it is hard to transition to a new mindset. However, I hope that this post will help some developers to see the value of Dependency Injection.
